Agent Infinite

Orquestrador local-first para múltiplos agentes CLI em terminais reais, organizados em um canvas infinito, onde o usuário conversa principalmente com um terminal orquestrador e este coordena agentes subordinados em outros terminais e worktrees.

Definição do produto

Agent Infinite é um app desktop open-source para orquestrar agentes de codificação como Claude Code, Codex, Gemini CLI, OpenCode e shells comuns em um canvas hierárquico de terminais reais.

Usuário ↓ conversa Orchestrator Terminal ├── Backend Agent Terminal ├── Frontend Agent Terminal ├── QA Agent Terminal └── Code Reviewer Terminal └── Security Agent Terminal

Posicionamento

  • Concorrente direto: OmniRift.
  • Referência de UX: Maestri + Figma.
  • Foco inicial: Windows-first.
  • Licença: MIT, 100% open source.
  • Meta inicial: 50 nodes por canvas.
Terminal-first

Agentes rodam em terminais reais, não como simples chamadas de API.

Orquestrador explícito

Todo canvas deve ter ao menos um node orquestrador.

Git/worktrees

Workspace vinculado a Git, com branches e worktrees desde cedo.

Event-based core

Elixir coordena comandos, eventos, estado e workers.

1. Core do produto

O core não é simplesmente “canvas com nodes”. O core é uma hierarquia operacional de terminais/agentes, comandada por um orquestrador.

core

Promessa central

  • Organizar agentes CLI em um canvas infinito.
  • Permitir hierarquias de papéis.
  • Centralizar a conversa no Orchestrator Terminal.
  • Delegar tarefas para agentes especializados.
  • Receber relatórios e resultados de volta.

Diferencial

  • Windows-first.
  • 100% open source MIT.
  • Terminal-first, sem depender de API keys.
  • Elixir event core local.
  • Worktrees Git como base do paralelismo.

Fora do foco inicial

  • SaaS multiusuário.
  • Marketplace.
  • Plugin system avançado.
  • Browser automation.
  • Sandbox rígido de segurança.

2. Workspace

Workspace é uma pasta/projeto/repositório Git. As configurações do Agent Infinite ficam dentro da pasta do próprio projeto.

core

Definição

  • Abrir pasta como workspace.
  • Exigir ou incentivar Git desde o início.
  • Detectar repositório Git automaticamente.
  • Criar config do Agent Infinite no projeto.
  • Carregar canvas e agentes ao reabrir.

Estrutura sugerida

meu-projeto/ .agent-infinite/ workspace.json canvas.json agents/ roles/ workflows/ git.json rules.json .gitignore

Ignorar no Git

logs/ sessions/ cache/ index/ tmp/ runtime.sqlite secrets.json terminal-buffers/ snapshots/local/

3. Orquestração hierárquica

O usuário conversa com o orquestrador. O orquestrador usa o mapa de agentes disponíveis para delegar, acompanhar e cobrar resultados.

core

Fluxo base

1. Usuário conversa com o Orchestrator Terminal.
2. Orchestrator identifica agentes filhos disponíveis.
3. Orchestrator delega tarefas aos agentes.
4. Agentes executam em terminais/worktrees próprios.
5. Resultados voltam para o Orchestrator.

Modelo hierárquico

  • Um node pode ser definido como orquestrador.
  • Nodes subordinados têm papéis especializados.
  • Um agente pode ter subagentes.
  • Conexões definem comando e reporte.
  • Todo canvas deve ter pelo menos um orquestrador.

Comando interno

O orquestrador pode usar uma CLI interna para delegar tarefas.

agent-infinite dispatch --to backend --task "Implemente autenticação JWT" agent-infinite ask --to qa --task "Teste o fluxo de login"

5. Conexões entre nodes

As conexões representam hierarquia, permissão de comunicação e fluxo de reporte. Não são apenas linhas visuais.

core

Tipos de relação

  • delegates_to: orquestrador delega tarefa.
  • reports_to: agente reporta resultado.
  • reviews: revisor avalia outro agente.
  • validates: QA/security valida entrega.
  • depends_on: agente depende de outro.

Controle inicial

  • Orchestrator sabe quem são seus filhos.
  • App sabe para qual terminal enviar a tarefa.
  • Elixir registra comandos e eventos.
  • Go escreve no PTY real do agente alvo.
  • Resposta volta como evento para Elixir.

Eventos estruturados

Command: DispatchTask Event: TaskDispatched Event: AgentOutputReceived Event: AgentCompleted Event: ResultReported

6. Terminais

Terminais são reais. O suporte inicial deve cobrir PowerShell, WSL e Git Bash no Windows.

core

Perfis iniciais

  • PowerShell como padrão Windows.
  • WSL como ambiente Linux separado.
  • Git Bash como perfil configurável.
  • Terminal por workspace.
  • Terminal por worktree.

Adendo importante

Cada terminal pode ter um ambiente diferente. WSL é outro sistema. Claude Code, Codex e outros agentes podem precisar ser instalados/logados novamente em cada ambiente.

Environment check

PowerShell: Git: ok Claude Code: ok Codex: ok WSL Ubuntu: Git: ok Claude Code: não instalado Codex: ok Git Bash: Git: ok Claude Code: ok Codex: não encontrado

8. Git e worktrees

Workspace deve estar vinculado a Git. Branches e worktrees são parte do core para isolamento dos agentes.

core

Ao abrir workspace

  • Detectar .git.
  • Ler branch atual.
  • Listar branches locais.
  • Puxar/listar branches remotas.
  • Listar worktrees existentes.
  • Exibir status do repo.

Worktree por agente

  • Criar worktree para tarefa/agente.
  • Abrir terminal naquela worktree.
  • Rodar agente isolado naquela branch.
  • Comparar diff entre worktrees.
  • Remover worktree ao finalizar.
  • Manter histórico do agente.

Controle futuro

  • Review de diff.
  • Aprovar/rejeitar entrega.
  • Merge controlado.
  • Rollback.
  • Resolver conflito com auxílio de agente.
  • Gerar relatório de mudanças.

9. Arquitetura

Elixir local coordena. Go Worker executa. UI renderiza e envia intenções para o core.

infra

Desenho atual

Electron + React UI ↓ WebSocket Elixir Core local ↓ protocolo local Go Worker local ↓ PTY / Git / WSL / PowerShell / Git Bash

Elixir Core

  • Fonte lógica de verdade do workspace ativo.
  • Gerencia nodes, edges e hierarquia.
  • Registra commands/events.
  • Orquestra Go Worker.
  • Distribui eventos para a UI.

Go Worker

  • Cria PTYs.
  • Executa comandos.
  • Controla processos.
  • Executa Git/worktrees.
  • Lê/escreve arquivos do workspace.
  • Envia outputs e status ao Elixir.

10. Persistência

Usar JSON para configuração versionável e SQLite para runtime/histórico.

core

JSON versionável

  • workspace.json.
  • canvas.json.
  • agents/*.json.
  • roles/*.json.
  • workflows/*.json.
  • rules.json.

SQLite local

  • Event log.
  • Terminal sessions.
  • Agent sessions.
  • Runtime state.
  • Logs consultáveis.
  • Histórico de execuções.

Regra

Configuração compartilhável e legível fica em JSON. Dados grandes, temporários, históricos ou sensíveis ficam fora do Git, preferencialmente em SQLite/cache local.

11. Canvas e performance

Canvas infinito conceitualmente, com meta inicial realista de 50 nodes por canvas.

produto

Meta inicial

  • 50 nodes por canvas.
  • 10 a 20 terminais/agentes ativos como teste inicial.
  • Pan e zoom suaves.
  • CPU baixa em idle.
  • Renderização sob demanda.

Estratégia

  • Sem render loop global contínuo.
  • Nodes fora da viewport simplificados ou dormindo.
  • Terminal minimizado não renderiza buffer completo.
  • Eventos de output com throttle/batching.
  • Evitar re-render global do canvas.

Stack sugerida

  • Electron + React + TypeScript.
  • React Flow ou engine híbrida.
  • xterm.js para terminais.
  • Zustand/Jotai para estado de UI.
  • Design system próprio estilo Figma.

13. Escopo de execução e segurança

No MVP, sem camada pesada de segurança. O app executa dentro do workspace, sem promessa de sandbox rígido.

mvp

Decisão atual

  • Sem approval gates obrigatórios no MVP.
  • Permitir execução e edição dentro do workspace.
  • Sem sandbox rígido inicialmente.
  • Sem bloquear comandos por padrão no MVP.

Limite realista

Como os terminais são reais, eles executam o que o sistema permitir. O app pode iniciar tudo no workspace, mas não deve prometer isolamento absoluto sem camada própria de sandbox/container.

Futuro

  • Modo seguro.
  • Approval gates.
  • Comandos bloqueados.
  • Logs auditáveis.
  • Sandbox por container.
  • Permissões por agente.

15. MVP funcional
